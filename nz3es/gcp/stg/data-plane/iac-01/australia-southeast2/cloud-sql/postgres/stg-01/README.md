## Users

| User | Role | Connection Limit |
|---|---|---|
| `users-owner` | Owner | 68 (75% of 90 usable) |
| `users-reader` | Read-only | 22 (25% of 90 usable) |

**Connection budget:** `max_connections = 100`, 10 reserved for internal/Cloud SQL admin = 90 usable.

Passwords are randomly generated by Terraform. Retrieve after apply:

```bash
terragrunt output
```

## Post-Apply: Set Per-User Connection Limits

Per-user `CONNECTION LIMIT` is a PostgreSQL role-level setting that cannot be managed by the Cloud SQL Terraform module. Run these SQL commands once after the initial `terragrunt apply`:

```sql
ALTER ROLE "users-owner"  CONNECTION LIMIT 68;
ALTER ROLE "users-reader" CONNECTION LIMIT 22;
```

Verify:

```sql
SELECT rolname, rolconnlimit
FROM pg_roles
WHERE rolname IN ('users-owner', 'users-reader');
```

> **Note:** This instance uses PSC with no public IP. Connect via Cloud SQL Auth Proxy or `gcloud sql connect`:
>
> ```bash
> gcloud sql connect <INSTANCE_NAME> --user=postgres --project=iac-01
> ```

## TODO: Automate Connection Limits via Terraform

The `ALTER ROLE` commands above are manual. Two options to manage this as IaC (source of truth):

### Option A: `postgresql` Terraform provider (recommended)

- Uses `cyrilgdn/postgresql` provider to manage `postgresql_role` with `connection_limit`
- Proper IaC — connection limits tracked in Terraform state
- Separate terragrunt unit (e.g. `stg-01-config/`) depending on `stg-01`
- `postgresql_role` with `skip_drop_role = true` — manages attributes only, doesn't own role lifecycle
- **Requires connectivity** to the DB from where Terraform runs:
  - From GitHub runner VM (in VPC) — connect directly via PSC endpoint
  - From local machine — run Cloud SQL Auth Proxy first

### Option B: `null_resource` + `local-exec`

- Runs `ALTER ROLE` via `psql` or `gcloud sql connect` as a Terraform provisioner
- Simpler but fragile — depends on local tools being available
- Harder to manage reliably in CI
- Not recommended
